# 260313 #35 执行结果与诊断文件分离

## 背景

Issue #35 要求把执行产物从“单文件混合承载”改成“结构化结果 + 独立诊断”双轨模型，避免 `patrol`、回帖和后续统计继续猜测 `result.json` 里到底是 JSON 还是错误文本。

上一轮 #34 为了保住 stderr 现场，把 `stdout+stderr` 一起送进了结果链路；这虽然补上了排障信息，但也把结果文件语义重新弄混了。

## 本轮实现

### 1. 执行器产物拆分

- 结构化结果文件继续使用 `var/results/<task>.json`
- 新增诊断文件 `var/results/<task>.diag.txt`
- 运行日志仍写入 `log/<task>.log`

同步执行时：

- 只在 stdout 成功解析为 Claude JSON 后写入结构化结果文件
- stderr 单独写入诊断文件
- 若 stdout 不是结构化 JSON，则清空结果文件，把 stdout/stderr 合并写入诊断文件

tmux 执行时：

- stdout 直接写结构化结果文件并追加到总日志
- stderr 直接写诊断文件并追加到总日志
- 若后续巡查发现结果文件不是合法结构化 JSON，会把无效内容迁移到诊断文件并清空结果文件，避免最终态继续污染结构化结果语义

### 2. 收口链路调整

- `Executor.LoadResult()` 先读结构化结果文件
- 当结构化结果缺失或解析失败时，回退诊断文件
- 若诊断文件也为空，才继续回退日志尾部
- 失败错误文案会明确写出“结构化结果缺失/解析失败，已回退诊断文件”，回帖不再把“无结果”和“有诊断”混成一类

### 3. 测试补齐

- 新增同步执行非 JSON stdout 时，结果文件清空、诊断文件保留原始输出的断言
- 新增 `LoadResult()` 读到无效结果文件后自动迁移到诊断文件的断言
- 新增 `patrol` 在结构化结果缺失时优先使用诊断文件的断言
- 保留结构化结果为空且诊断文件为空时，再回退日志尾部的断言

## 验证

已执行：

```bash
go test ./internal/executor ./internal/app ./internal/adapters/reporter
```

## 结果

- 结构化结果文件最终态只保留机器可解析 JSON
- 诊断文本有稳定独立落点，不再必须依赖 tmux pane 现场
- `patrol` / `reporter` 能明确区分“结果缺失”和“诊断存在”

## 后续观察点

- tmux 路径当前仍允许执行期把 stdout 先落到结果文件，最终由收口阶段做语义纠偏；如果后续要把“执行中间态”也完全收紧，需要再补一个更强的 post-process 归档动作
- 统计/审计链路若要直接消费诊断文件，可继续在后续单里补 machine-readable 的产物索引
