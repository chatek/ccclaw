## 代码路径核查

### 1. 当前门禁实现位置

核心逻辑位于：

- `src/internal/app/runtime.go`
- `src/internal/adapters/github/client.go`

其中 `syncIssue()` 的判定路径是：

1. 调 `client.IsTrustedActor(issue.User.Login, minimumPermission)`
2. 若发起者权限达到最小门槛，则先将 `approved = true`
3. 再读取最新评论中的 `/ccclaw <批准词|否决词>`
4. 否决词可覆盖为 `false`
5. 批准词可覆盖为 `true`

也就是当前真实语义是：

- “可信发起者自动放行”
- “评论审批是补充覆盖项，不是唯一入口”

### 2. 关键代码事实

在 `src/internal/app/runtime.go` 中，当前逻辑等价于：

1. 先按发起者权限判定 `trusted`
2. `approved := trusted`
3. 若最新评论是否决，则改为 `false`
4. 若最新评论是批准，则改为 `true`

换言之，只要满足以下条件：

- Issue 是 open
- 带 `ccclaw` 标签
- 作者权限达到 `approval.minimum_permission`，默认是 `maintain`

即使没有任何批复评论，也会自动进入可执行状态。

### 3. 当前实现与现有文档是一致的

仓库内多处文档与配置注释当前都写的是：

- `maintain` 及以上权限的 Issue 自动执行
- 其他 Issue 需要受信任成员评论 `/ccclaw <批准词>`

Issue #27 的拍板与已落地实现也延续了这条规则，并没有引入“管理员也必须显式评论批准”的新约束。

因此，这次行为从“现有实现是否偏离已拍板规则”来看，答案是：

- 没有偏离
- 当前行为符合现行实现，也符合现行文档

## 根因判断

这次现象的根因不是审批解析错误，而是规则边界不足。

更准确地说，当前系统只区分了两类 Issue：

1. 可信成员直接发起
2. 非可信成员发起，需额外批复

但当前实际协作里已经出现第三类：

3. 可信成员身份下，由代理或自动化工具代发、仅用于讨论/记录的 Issue

对系统来说，第 1 类和第 3 类目前没有任何可区分信号：

- 作者账号相同
- 仓库权限相同
- 同样带 `ccclaw` 标签

于是运行时只能把第 3 类也当成“可信成员主动投递任务”，直接放行执行。

## 结论

本次问题的结论应分成两层：

### 1. 现象层结论

Issue #28 之所以在没有 `/ccclaw go` 的情况下仍开始执行，直接原因是：

- 作者 `ZoomQuiet` 在 `41490/ccclaw` 中被动态判定为 `admin`
- Issue 本身带有 `ccclaw` 标签
- 当前规则就是“`maintain` 及以上发起者自动执行”

### 2. 规则层结论

当前门禁缺少“代理代发 Issue”与“管理员明确投递任务”之间的区分机制。

所以这不是一个“代码算错了批准状态”的 bug，而是一个“现有规则不足以覆盖新协作方式”的缺口。

## 建议的收口方案

由于这已经涉及门禁语义调整，不宜在未拍板前直接修改实现。建议按“先止血、后改规则”的顺序处理。

### A. 立即止血：调整 Issue 创建习惯

从现在开始，使用 `gh` 创建“讨论型 / 调研型 / 记录型” Issue 时：

- 不要主动带上 `ccclaw` 标签

原因很直接：

- 在当前规则下，`ccclaw` 标签本身就是执行入口信号
- 一旦作者又是 `maintain` 及以上，就会自动进入执行

也就是说，在现行规则不变的前提下：

- “不带 `ccclaw` 标签创建讨论 Issue” 是最小且立刻生效的止血方式

### B. 规则改造候选方案

后续建议在 Issue 中二选一拍板：

#### 方案 1：保留“可信作者自动执行”，新增显式抑制信号

例如增加一种明确的阻断机制，满足任一条件即不自动执行：

- Issue 带某个保留标签，例如 `discussion` / `draft`
- Issue body 含固定前缀字段，例如 `execution: manual`
- Issue 由固定自动化账号创建时默认仅巡查不执行

优点：

- 对现有“管理员快速投递任务”习惯改动最小
- 不会破坏现有自动执行能力

缺点：

- 需要额外设计并固化“抑制信号”的契约

#### 方案 2：收紧规则，改为所有 Issue 都必须显式评论批准

即改成：

- 无论作者权限高低
- 只要要执行，就必须出现受信任成员的 `/ccclaw <批准词>`

优点：

- 语义最清晰
- 讨论 Issue 与执行 Issue 的区分最明确

缺点：

- 会推翻当前“可信作者自动执行”的既有规则
- 需要同步修订 README、配置注释、已有使用习惯与相关测试

## 我的建议

我建议优先采用“方案 1”而不是直接推翻现有规则。

原因：

1. 当前仓库已多次明确写入“maintain 及以上自动执行”
2. 这条规则本身并没有错，只是现在新增了“代理代发讨论 Issue”的场景
3. 相比整体收紧，给自动化代发场景补一个显式抑制信号，改动面更小，也更易验证

建议的默认收口顺序：

1. 立刻约束操作习惯：
   - 以后用 `gh issue create` 创建讨论型 Issue 时，不带 `ccclaw` 标签
2. 在后续 Issue 中拍板一个“自动执行抑制信号”
3. 拍板后再落代码、补测试、同步文档

## 本轮处理结论

本轮仅完成调查、结论澄清与方案建议，不直接修改门禁实现。

原因是这里已经出现明确规则分歧：

- 当前文档支持自动执行
- 当前协作预期希望“代理代发的管理员 Issue”不要自动执行

按仓库约束，这类分歧应先回 Issue 拍板，再落代码。
